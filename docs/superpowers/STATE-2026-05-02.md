## Critical accumulated knowledge

### bootFlag (RAM 0x00F9) — DEPRECATED HEURISTIC

V1 used `bootFlag != 0x4D → TitleOrMenu`. Diagnostic (`BootFlagDiagnosticTest`) proved the FF1 ROM sets bootFlag to 0x4D within ~9 frames of cold boot, regardless of input. So the heuristic falsely classified almost everything as Overworld(0,0).

V2 fix in PR #94: dropped bootFlag entirely. Real markers: `char1_hpLow != 0` (party created), `worldX/Y != 0` (on world map), `screenState == 0x68/0x63` (battle/post-battle), `locationType == 0xD1` (indoors), all 4 `char_status` bit-0 set + party-was-created → PartyDefeated.

### Koog 0.6.1 surface (canonical)

- `singleRunStrategy()` lives in `ai.koog.agents.core.agent` (not `agents.ext`). Takes `ToolCalls` enum, no `name` param.
- **Important:** `singleRunStrategy` does NOT mean "1 LLM call per run". It means "1 graph run" — Koog still loops while the model returns tool calls. Without `maxIterations` cap on AIAgent, model loops up to 50 times.
- Our cap: `maxIterations = 10` for executor, `8` for advisor. Catch handler treats cap-fired exception as a normal turn outcome.
- Koog has **no Anthropic prompt-caching support** in 0.5.1 or 0.6.1 (Path B per cache probe). Future V2.x can swap in a custom HttpClient with `Anthropic-Beta: prompt-caching-2024-07-31` headers.
- Reflection-based `tools(toolset)` rejects `Map<String, String>` parameters. `EmulatorToolset.executeAction` had its `args` param removed.

### FF1 ROM behavior on this codebase

- Default party from A-mash: FIGHTER, THIEF, BLACK BELT, RED MAGE (verified by RAM stats matching FF1 starting class stats).
- After `pressStartUntilOverworld`, party starts at world coords `~(0x92, 0x9E)` = `(146, 158)`.
- Walking N from (146, 158) reaches (146, 152) (6 tiles), then BLOCKED. Greedy walk doesn't path-find — keeps trying same direction. **This is the current navigation deadend.**

### V2.1 wrong hypothesis, V2.2 wrong fix

V2.1 trace showed agent stuck at (146, 152) and the model itself hypothesised "stuck inside Coneria castle". V2.2 added `Indoors` phase + `exitBuilding` skill to address it.

But V2.2 live re-run still classifies the stuck position as `Overworld(146, 152)` — `locationType != 0xD1` there. The party is genuinely on the **overworld**, just surrounded by impassable terrain (FF1 Coneria area is an island; mountains/ocean block direct N from this tile).

V2.2 commit is **valid infrastructure** for the day the agent enters a town/castle, but does not fix today's bug.

## Real V2.3 problem

**Spatial reasoning gap.** LLM has only RAM coords (numeric). It doesn't know "(146, 152) is bordered by mountains north / water west." Greedy `walkOverworldTo` keeps trying north, fails. Random `walkUntilEncounter` doesn't help (also blocked).
